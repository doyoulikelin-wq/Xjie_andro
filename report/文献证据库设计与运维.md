# Xjie 文献证据库（Literature Evidence Base）设计与运维手册

> 版本：v1.0 ｜ 日期：2026-04-19
> 适用范围：Xjie 后端 `app/services/literature/*`、`/api/literature/*`、AI 对话 RAG 注入

---

## 0. 设计目标

为 AI 对话「小捷」在给出**干预/结论**时，自动附带带证据等级的文献引用：

- **范围**：首期 500 条结论，覆盖
  - (a) CGM × 食物 × 餐后血糖响应（PPGR）
  - (b) 代谢组学 × 慢病风险（糖尿病 / MASLD / 心血管 / 老龄化）
- **合规**：仅存储我们自行撰写的中文一句话结论 + 文献元数据，**不复制摘要原文**
- **证据分层**：L1（Meta/系统综述/大型 RCT）> L2（队列 >1000/小型 RCT）> L3（病例对照/机制）> L4（综述/共识）
- **软约束**：覆盖率不足时由 LLM 通用知识兜底，不强制必须命中
- **展示形态**：气泡正文小角标 `[N]` + 底部「参考文献」列表 + 点击弹窗展开详情

## 1. 数据库结构

### 1.1 实体关系

```
literature ────< literature_claim ────< literature_claim_concept
                       │
                       └── embedding (JSON list[float])
literature_ingest_job  (审计 / 增量记录)
```

### 1.2 表说明

| 表 | 关键字段 | 说明 |
|----|---------|------|
| `literature` | pmid (UNIQUE), doi, title, authors[], journal, year, language, **evidence_level (L1-L4)**, study_design, sample_size, population, **conclusion_zh (REQUIRED)**, conclusion_en, topics[], source, fetched_at, reviewed, reviewer | 文献元信息。**不存摘要原文**，只保留我们的一句话中文结论 |
| `literature_claim` | literature_id (FK), claim_text, claim_text_en, exposure, outcome, effect_size, direction, confidence, topics[], tags[], evidence_level (反范式), embedding (JSON), embedding_model, enabled | 单条可检索结论。一篇文献可拆 1-3 条 claim |
| `literature_claim_concept` | claim_id (FK), role (exposure/outcome/population), concept_key, concept_label | 概念词典，用于跨语言/同义词匹配（首期可空）|
| `literature_ingest_job` | query, topic, status, fetched/inserted/skipped, error, meta, created_at | 抓取任务审计，幂等去重依据 |

### 1.3 跨数据库兼容

`app/db/compat.py` 提供：

- `JSONB` → PG 上是 JSONB，SQLite 上降级为 `JSON`
- `StringArray` → PG 上是 `ARRAY(String)`，SQLite 上为 `Text`（逗号分隔）

迁移：`backend/app/db/migrations/versions/0010_literature.py`，`down_revision="0009_feature_flags_skills"`。

## 2. 数据流水线

```
PubMed E-utilities (esearch + efetch)
        ↓
   PubMedRecord (in-memory, 含 abstract)
        ↓                                 ┌─ infer_evidence_level (publication_types 启发式)
        ↓                                 │
   LLM claim_extractor (Kimi/OpenAI) ────┤   输出 JSON：skip + 1~3 claims + study_design + sample_size
        ↓                                 │   严格规则：claim_text ≤80 字、自撰中文、effect_size 必须真实、不复制摘要 >10 字
        ↓
   embed_text(claim) → vector
        ↓
   Literature + Claim 入库（不写入 abstract）
        ↓
   IngestJob 审计行
```

关键模块：

- [backend/app/services/literature/pubmed_fetcher.py](backend/app/services/literature/pubmed_fetcher.py) — `httpx` 实现，含 `infer_evidence_level`
- [backend/app/services/literature/claim_extractor.py](backend/app/services/literature/claim_extractor.py) — Kimi K2.5 兼容，`response_format=json_object`，无 LLM 时返回 `skip=True`
- [backend/app/services/literature/embedding.py](backend/app/services/literature/embedding.py) — `text-embedding-3-small` (1536d) ；Kimi base_url 时降级为本地 256d hash 伪向量
- [backend/app/services/literature/ingest.py](backend/app/services/literature/ingest.py) — 全流程幂等 (按 PMID 去重)

## 3. 检索服务（RAG）

### 3.1 混合打分

```
score = max(cosine(q_vec, claim_vec), 0) + KEYWORD_BOOST × (#matched_keywords)
threshold = 0.30, KEYWORD_BOOST = 0.18
```

- 关键词抽取：英文小写词 + 中文 1~2 字 bigram，剔除停用词
- `_evidence_floor("L2")` → `{"L1","L2"}`，可同时拿到更高等级
- 全量加载已启用 claims 进内存（500 条无压力）

> 当语料 >5K 条时，建议切到 `pgvector` + ivfflat 索引：仅替换 `retrieval.py` 内的候选召回阶段，混合打分逻辑可复用。

### 3.2 软约束

- 命中 → 把 `[1] short_ref (level, n=X) — claim_text` 注入 system prompt 末尾，并指示 LLM 在涉及结论时使用 `[N]` 角标
- 未命中 → 不注入任何引用，让 LLM 走通用知识

### 3.3 路由

| Method | Path | 用途 |
|--------|------|------|
| POST | `/api/literature/retrieve` | 用户/调试：按 query 拉引用 |
| GET | `/api/literature/claims/{id}/citation` | iOS 点击角标展开 |
| GET / POST | `/api/literature/items` | Admin：文献列表 / 新建 |
| GET / POST / PATCH | `/api/literature/claims` | Admin：结论 CRUD |
| POST | `/api/literature/ingest` | Admin：触发抓取（≤100 条/次）|
| POST | `/api/literature/reembed` | Admin：全量重新 embed |
| GET | `/api/literature/jobs` | Admin：审计 |
| GET | `/api/literature/stats` | Admin：按 level/topic 统计 |

权限：用户接口 `get_current_user_id`，管理接口 `require_admin`。

## 4. AI 对话集成

[backend/app/routers/chat.py](backend/app/routers/chat.py) 的同步 + SSE 两条链路均已注入：

1. 解析 user message
2. `retrieve_claims(db, query=msg, top_k=5)`
3. 命中则在 skill prompt 末尾追加：

   ```
   --- 文献证据（请在正文用 [N] 引用，结尾给"参考文献"区块）---
   [1] Segal et al., Cell 2015 (L1, n=800) — 个体餐后血糖差异最高 8 倍...
   ...
   ```
4. 持久化 `ChatMessage.meta.citation_ids = [...]`
5. 响应 `ChatResult.citations` / SSE `done` 事件附带 `citations`
6. `GET /conversations/{id}` 时按 ID 批量回填 → iOS 历史消息也能看到引用

## 6. 运维 SOP

### 6.1 启用迁移

```bash
cd backend
source .venv/bin/activate
alembic upgrade head    # 0010_literature 创建 4 张表
```

### 6.2 首期种子抓取（约 500 条）

```bash
# 9 个高质量 PubMed 检索串，详见 backend/app/workers/literature_seeds.json
python -m app.workers.literature_ingest --seed app/workers/literature_seeds.json
```

种子覆盖：CGM PPGR / 个性化营养 / GI 食物 / 代谢组学+T2D / 代谢组学+MASLD / 代谢组学+心血管 / 代谢年龄时钟 / 地中海饮食 / 限时进食。

### 6.3 单条检索补充

```bash
python -m app.workers.literature_ingest \
  --query "(personalized nutrition[Title]) AND (CGM OR continuous glucose monitoring)" \
  --topic ppgr --max 50 --min-year 2018
```

### 6.4 切换到 OpenAI embedding

默认用 Kimi 时为 256d hash 伪向量。如果接入 OpenAI，设置：

```
LITERATURE_EMBEDDING_MODEL=text-embedding-3-small
OPENAI_BASE_URL=https://api.openai.com/v1
OPENAI_API_KEY=sk-...
```

然后：

```bash
curl -X POST -H "Authorization: Bearer $ADMIN_TOKEN" \
  http://127.0.0.1:8000/api/literature/reembed
```

### 6.5 人工审核标记

抓取后建议人工抽审：

```bash
PATCH /api/literature/claims/{id}    body: {"enabled": true}
PATCH /api/literature/items/{id}     body: {"reviewed": true, "reviewer": "lin@xjie"}
```

仅 `enabled=true` 的 claim 会被检索召回。

## 7. 合规边界

- ❌ 不存储 / 不返回 PubMed 摘要原文（claim_extractor 已硬规则限制 ≤10 字符摘录）
- ✅ 存储自撰中文一句话结论 + 题录信息（合理使用）
- ✅ 角标默认不跳转外站；如需跳转 PubMed，可后续以独立模块开关控制
- ✅ 用户态接口仅暴露 `CitationBundle`（不含完整 abstract、不含 LLM 抓取的 raw 数据）
- ⚠️ 干预性建议（用药、剂量）应只引用 L1+L2；对话 prompt 已要求 LLM 优先使用更高等级证据
